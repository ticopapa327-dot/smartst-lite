# 桌面版手术示教软件开发计划

> 适用仓库：`D:\我的工作\AOV\SmartST Lite`。  
> 当前基线：Tauri 2 + React + TypeScript + Rust，当前主线为 USB-first 工作台、LiveKit UI PoC、真实 JWT smoke、Native Worker 控制面、USB 通道绑定和安装版 smoke；旧 ONVIF/RTSP MVP 已从当前产品入口移除。
> 计划原则：先验证硬件和媒体链路，再做完整 UI；先保证本地预览和录像，再接远程互动；SmartST Server、SmartST OR Agent、Desktop Client 三包逻辑分离；患者信息和文件管理从第一天按敏感数据处理。

## 1. 目标版本定义

### V0.1 技术验证版

目标：证明 Windows 桌面端可以完成 4 路本地预览、1 路远程 LiveKit 互动、基础音频通话。

不承诺：

- 长时间医疗录像稳定性。
- HIS 正式对接。
- 多医院部署。
- 所有 USB PTZ 摄像机兼容。

### V0.2 可演示版

目标：手术室端和示教室端能完成呼叫、应答、仅收看/交互模式、主画面远程观看、双向语音、单路录像、基础文件列表。

### V0.3 院内试点版

目标：支持 4 路预览、按通道录像、患者绑定、导出、上传、标注同步、多显示器扩展、会议人数限制、Android 会议平板客户端、手机 H5 单向收看。

### V1.0 可交付版

目标：完成现场硬件白名单、长时间录像、断网/断电恢复、审计、权限、部署脚本、运维文档和验收测试。

## 2. 阶段计划

## P0 需求冻结与硬件验证

周期：1 到 2 周。

交付物：

- 硬件清单：摄像机、采集卡、USB Hub、全向麦、声卡、显示器。
- UI 视觉基线确认：采用 `or-preview HMI palette v0.3`，不得另起蓝色科技屏或 BI 风格。
- 4 路 USB 采集压力测试记录。
- PTZ 控制协议确认。
- SmartST Server 一体机/分机部署模式确认。
- LiveKit 本地/院内服务节点连通性测试。
- HIS 接口方式确认：HL7/FHIR/WebService/DB/手工。

关键任务：

- 枚举 Windows 摄像头、麦克风、扬声器。
- 验证 WebView2 是否能同时打开 4 路视频。
- 验证 1080p30、1080p60、4K 降级。
- 验证全向麦 + 扬声器 AEC 效果。
- 验证 LiveKit 房间发布主画面和麦克风。
- 验证 RTSP 不能直接进入 WebView/LiveKit，需要 FFmpeg/GStreamer 转接。
- 确认 FTP 是否必须是明文 FTP；若不是强制，改用 SFTP/FTPS。
- 将 `docs/ui-visual-style.md` 的 CSS token 写入前端样式基线。
- 将 `docs/deployment-package-split.md` 作为安装器、服务化和目录迁移的约束基线。

验收标准：

- 4 路本地画面预览 30 分钟不黑屏、不崩溃。
- LiveKit 主画面 + 双向语音通话 30 分钟可用。
- 明确每个 PTZ 设备的控制方式和不可控项。
- UI 原型符合冷灰蓝 HMI 视觉，不使用大面积蓝色、紫蓝渐变、玻璃拟态。
- 输出《硬件兼容性与风险清单》。

失败门槛：

- 如果 WebView2 无法稳定 4 路采集，立即切换 Native Media Worker 方案，不继续在前端堆补丁。
- 如果全向麦 AEC 不稳定，必须限定推荐硬件，不能靠软件承诺。

## P0.5 三包服务化基线

周期：1 到 2 周。

目标：把当前 PoC 从“桌面端直接托管所有后端能力”调整为 SmartST Server、SmartST OR Agent、Desktop Client 三个部署单元。

交付物：

- `SmartST Server` 原型：封装当前 `server-poc`，明确 LiveKit Server 配置、API secret 存储、RoomService preflight 和业务服务端口。
- `SmartST OR Agent` 原型：封装 Native Worker 生命周期、设备枚举、start/status/stop/drain 控制面，预留 Windows Service。
- Desktop Client 调用边界：UI 通过 Server API 和 OR Agent API 工作，不直接保存 API secret，不直接承担长时间服务生命周期。
- 安装器角色设计：Server、OR Agent、Desktop Client 可单独选择，手术室电脑默认三项同装。
- 一体机 preflight：LiveKit、业务服务、OR Agent、Native Worker、端口、防火墙、默认 room。

关键任务：

- 定义 `packages/contracts` 或等价共享合同：endpoint、room、mediaPolicy、token grants、agent status、recording status。
- 将 `server:poc:livekit-preflight` 扩展为可检查本机一体机环境。
- 设计 Windows Service 安装方式，优先评估 WinSW 或原生 service wrapper。
- 将 Tauri 后端中 Native Worker session 管理边界迁移为 OR Agent API 设计。
- 明确 `C:\ProgramData\SmartST\` 配置、日志、密钥、录像索引和权限。

验收标准：

- 关闭 Desktop Client 后，Server 和 OR Agent 仍可继续运行。
- API secret 只存在 Server 配置或服务环境中。
- OR Agent 可在没有 UI 的情况下完成设备枚举和短时采集 smoke。
- 一体机部署不要求专用服务器，但进程、端口、日志和升级边界清晰。

## P1 本地工作台与采集层

周期：2 到 4 周。

交付物：

- 手术室工作台：默认 4 路预览。
- 视频源管理：USB 摄像机/USB 采集卡为默认入口，RTSP/SRT 仅作为后续高级输入适配器预留。
- 音频设备选择：输入、输出、音量、电平。
- PTZ 控制面板：能力检测、移动、缩放、聚焦、预置位。
- 设备别名和通道角色保存。

关键任务：

- 新增领域模型：`VideoSource`、`AudioDevice`、`PtzDevice`、`ChannelLayout`。
- 新增服务：`deviceService`、`captureService`、`audioService`、`ptzService`。
- UI 改为工作台优先，不再把 ONVIF/RTSP 表单作为主入口。
- 实现热插拔提示和设备占用错误。
- 增加降级策略：分辨率、帧率、关闭非关键预览。

验收标准：

- 插入 4 路采集设备后，软件可识别、命名、保存角色。
- 关闭软件再打开，通道角色和设备偏好仍存在。
- 任一设备断开，界面给出明确提示，不影响其他通道。
- PTZ 不支持的设备不会显示误导性控制按钮。

## P2 LiveKit 呼叫与互动

周期：3 到 5 周。

交付物：

- SmartST Server 最小版：端点注册、呼叫、应答、拒绝、token 签发、LiveKit preflight。
- LiveKit 接入：房间连接、发布主画面、订阅远端画面、双向语音。
- 仅收看/交互模式。
- 参与人数上限。
- Android 会议平板客户端接入。
- 手机 H5 观察者只读接入和独立并发上限。
- 按需请求其他通道。

关键任务：

- 增加 `livekit-client`。
- SmartST Server 签发短期 token，客户端和 OR Agent 不保存 LiveKit API secret。
- 建立手术室端在线注册和示教室端呼叫流程。
- 使用 LiveKit 权限区分 `canPublish`、`canSubscribe`、`canPublishData`。
- 实现示教室仅收看模式：不能发布麦克风/摄像头。
- 实现交互模式：允许示教室发布麦克风。
- 实现人数限制：业务服务检查 + LiveKit room 限制。
- 实现 Android 会议平板客户端注册和 `tablet-watch` / `tablet-interactive` token 策略。
- 实现手机 H5 观察者 token：只能订阅默认画面，不能发布音视频或控制消息。
- 验证手机多并发由 LiveKit/SFU 转发，手术室端只发布一次默认画面。
- 增加网络质量、断线重连、token 过期提示。

验收标准：

- 示教室输入手术室 IP/端点后可发起呼叫。
- 手术室可接受或拒绝，接受时可确认仅收看/交互。
- 双方建立连接后默认显示一个画面，语音稳定。
- 示教室请求第二/第三/第四路画面时，手术室确认后可见。
- Android 会议平板可作为正式终端加入，仅收看/交互权限按 token 生效。
- 手机端通过 H5 链接只能单向收看默认画面，不能发起交互。
- 10 个手机端同时观看时，手术室端上行不按 10 路增长。
- 超过人数限制的新用户收到明确提示。

## P3 录像、回放、导出和上传

周期：4 到 6 周。

交付物：

- 按通道选择录像。
- 本地录像文件和 manifest。
- 回放列表和检索。
- 移动存储导出。
- FTP/SFTP/FTPS 上传任务。
- 基础异常恢复。

关键任务：

- 定义录像目录结构。
- 实现 SQLite 索引。
- 实现单通道 MP4 录制。
- 实现合成画面录制。
- 实现录像开始/停止/失败状态机。
- 磁盘空间不足前预警。
- 导出时写入 manifest 和校验文件大小。
- 上传任务支持重试、失败原因、人工重传。

验收标准：

- 可选择 1 到 4 路进行录像。
- 录制 2 小时后文件可正常播放。
- 录制期间某一路断开，不导致全局崩溃，并在 manifest 记录。
- 可按日期、手术号、患者绑定查找录像。
- 可导出到 U 盘，导出完成后校验通过。
- 上传失败可重试，失败原因可读。

## P4 HIS 绑定和文件管理

周期：2 到 4 周。

交付物：

- HIS 查询适配器。
- 手术/患者信息选择与绑定。
- 脱敏显示策略。
- 文件管理页。
- AI 处理接口预留。

关键任务：

- 定义 `PatientBinding` 数据模型。
- 支持手工绑定作为兜底。
- 根据院方接口实现一个 adapter。
- 录像 manifest 写入绑定 ID，不把完整患者信息写入文件名。
- 日志脱敏。
- 文件管理页预留 AI job 状态。

验收标准：

- 手术室端可搜索/选择患者或手术排班。
- 新录像可绑定患者信息。
- 示教室默认只看到脱敏信息。
- 导出文件不包含敏感明文文件名。
- AI 预留接口可创建 job 记录，但不要求模型处理。

## P5 标注、多显示器和会议模式

周期：3 到 5 周。

交付物：

- 手术室端标注。
- 远端同步标注。
- 多显示器独立窗口。
- 可调整布局。
- 会议模式主持控制。

关键任务：

- 实现标注 Canvas/SVG 层。
- 标注事件通过 LiveKit data/RPC 发送。
- 多显示器：Tauri 多窗口或原生窗口绑定 track。
- 布局保存：网格、主辅、单画面全屏。
- 主持控制：静音、移除、升级交互权限。

验收标准：

- 手术室端标注在示教室端 300 ms 内可见。
- 单个画面可扩展到指定 Windows 显示器。
- 关闭再打开后恢复上次布局。
- 主持端可限制参会人数和交互权限。

## P6 安全、部署和验收

周期：3 到 6 周。

交付物：

- 院内部署文档。
- SmartST Server / SmartST OR Agent / Desktop Client 角色安装脚本。
- Windows Service 安装、升级、卸载和防火墙规则脚本。
- 日志与审计。
- 安全配置基线。
- 验收测试报告。

关键任务：

- TLS 证书和内网域名配置。
- LiveKit 服务部署和端口说明。
- TURN 配置和弱网测试。
- Windows 安装包签名。
- 崩溃日志、媒体诊断、网络诊断。
- 权限和审计记录：登录、呼叫、录像、导出、上传、权限变更。

验收标准：

- 新机器按文档 30 分钟内完成客户端安装。
- 服务端重启后，客户端能恢复可用状态。
- 断网/恢复后，系统有明确状态提示。
- 审计能追踪谁发起呼叫、谁开始录像、谁导出文件。
- 不在日志中泄漏 LiveKit secret、HIS 凭据和患者完整敏感信息。

## 4. 人员配置

最低配置：

- 1 名 Windows/Tauri 前端工程师。
- 1 名音视频工程师，熟悉 WebRTC、FFmpeg/GStreamer、采集卡。
- 1 名后端工程师，负责业务服务、LiveKit token、HIS、审计。
- 1 名测试/实施工程师，负责硬件矩阵、现场部署、验收。

风险提示：如果没有音视频工程师，不建议承诺 4 路长时间稳定录像和 PTZ 兼容。

## 5. 技术债控制

必须禁止：

- 客户端硬编码 LiveKit API key/secret。
- 用 UI 状态假装已经连接 LiveKit。
- 明文保存 HIS 凭据、摄像机密码、FTP 密码。
- 把患者姓名、身份证号写入录像文件名。
- 只做合成录像，不保留单通道原始录像。
- 把 RTSP 当成 LiveKit 原生能力。
- 在未验证硬件前承诺 4K 多路稳定。

可以接受的临时债：

- 演示版先用 WebView2 `MediaDevices`。
- 演示版先只发布一路主画面。
- 演示版 HIS 使用 mock 或手工录入。
- 演示版录像先做单路，明确标注不是正式医疗归档能力。

## 7. 验收清单

手术室端：

- 默认显示 4 路本地预览。
- 可选择音频输入输出。
- 摄像机支持 PTZ 时可控制，不支持时不误导。
- 可选择部分或全部通道录像。
- 可绑定患者信息并开始录像。
- 可接受或拒绝示教室呼叫。
- 可设置仅收看/交互/会议人数。
- 可标注画面并同步到远端。

示教室端：

- 可通过 IP/端点呼叫手术室。
- 可选择仅收看或交互模式。
- 连接后默认显示一个画面并建立语音。
- 可请求其他通道。
- 可调整布局。
- 可将单个画面扩展到指定显示器。
- 人数超限时收到明确提示。

服务端：

- token 短期签发。
- LiveKit secret 不下发到客户端。
- 呼叫、权限、房间、人数限制可审计。
- Android 会议平板有独立客户端类型和权限策略。
- 手机 H5 观察者有独立并发上限，只能单向收看。
- HIS 查询和录像绑定可追踪。
- 上传、导出、删除均有记录。

## 8. 第一轮实际开发建议

第一轮不要直接做全量系统。建议只做 4 个最小闭环：

1. 4 路 USB 预览闭环。
2. 主画面 LiveKit 发布/订阅闭环。
3. 双向语音 + AEC 闭环。
4. 单路录像 + manifest 闭环。

这 4 个闭环通过后，再进入 HIS、FTP、会议、标注、多显示器。否则系统会在最底层媒体能力未确认时过早复杂化。
