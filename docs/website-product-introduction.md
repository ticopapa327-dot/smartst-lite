## 产品定位

SmartST Lite 是面向 Windows 手术室终端的轻量级手术示教软件。新版本不再把网络摄像机、ONVIF、RTSP 地址和编码器推流配置作为默认使用路径，而是围绕 USB UVC 摄像机和 HDMI/SDI 转 USB 采集卡设计主流程。

产品目标是降低手术示教的视频接入门槛。手术室端优先完成本地多路预览、本地录像、默认画面选择和远端互动接入；远端观看和多人并发由业务服务与 LiveKit/SFU 体系承担，避免手术室终端为每个手机观察者单独推流或转码。

SmartST Lite 适合作为手术教学、观摩、术中讨论、院内培训和轻量化示教直播的客户端基础，不是完整数字化手术室平台，也不是诊断、治疗决策、生命支持、麻醉监护或医疗器械控制系统。

## 适用场景

- 手术室到示教室的实时观摩和双向语音互动。
- 医美、民营医院、动物医疗机构等轻量化手术教学场景。
- 内镜、超声、监护仪、显微镜、PACS 工作站等画面的 USB 采集预览。
- 院内会议平板的大屏观看、讨论和互动。
- 手机 H5 临时单向观看，例如观摩、巡查、家属沟通等只读场景。
- 单路或多路录像、导出、文件检索和后续归档系统对接。

## 核心能力规划

以下能力可以作为网站卖点提前宣传，建议在详情页用“规划支持”“试点支持”“可按项目交付”区分成熟度。

### USB 采集优先

软件主流程围绕免驱 USB UVC 摄像机和 HDMI/SDI USB 采集卡设计。术野主画面、内镜、超声、监护仪和辅助画面规划支持按通道接入、命名和保存角色。RTSP、SRT、ONVIF 等网络视频源保留为高级兼容能力，不再作为默认入口。

### 手术室本地工作台

手术室端采用 Windows 桌面客户端，第一屏直接进入工作台，展示视频源状态、主画面、辅助画面、直播状态、录像状态和远端连接状态。产品路线强调本地预览和本地录像不依赖远程连接，远程服务异常不应影响本地采集与录像。

### 实时互动和权限控制

远程互动由业务服务和 LiveKit/SFU 体系负责。业务服务规划负责端点注册、呼叫、应答、拒绝、房间创建、短期 token 签发、人数限制和审计；LiveKit/SFU 负责实时音视频转发、房间通信、数据通道和弱网传输。客户端不保存 LiveKit API secret。

### 默认画面和按需通道

远端连接后默认只发布或订阅一路默认画面。默认画面由手术室端业务规则决定，不依赖设备排序或媒体轨道自动排序。示教室如需查看其他通道，规划通过按需请求机制，由手术室端授权后再发布或订阅对应画面。

### 示教室和会议平板

Windows 示教室客户端用于正式示教互动，可根据权限进入仅收看或交互模式。Android 会议平板作为正式客户端规划接入，可用于横屏大屏观看、语音互动和会议场景，不等同于手机 H5 观察端。网站可以把 Android 会议平板写成试点规划能力。

### 手机 H5 单向观察

手机端不安装客户端，只作为 web-observer 单向收看。手机 H5 默认只能订阅手术室默认画面和音频，不允许发布麦克风、摄像头、标注、PTZ 或控制消息。手机并发由 LiveKit/SFU 承担媒体转发，手术室端只发布一次默认画面。

### 录像、导出和文件管理

录像能力规划为本地 Native Media Worker 承担采集、编码、录像和异常恢复。系统路线支持单通道录像、合成画面录像、manifest 元数据、导出校验、失败重试和文件索引。患者信息只用于绑定和检索，不写入录像文件名或日志明文。

### HIS 和患者信息边界

HIS 对接作为院内试点阶段能力规划，可以宣传为“支持按院方接口方式对接手术排班、患者绑定和录像索引”。患者信息按敏感数据处理，默认不向远端示教室展示完整身份信息。需要展示时应有配置开关、权限控制和审计记录。手工绑定作为无 HIS 接口场景的兜底方式。

### 安全和部署

推荐院内局域网部署业务服务、LiveKit/SFU 和必要的 TURN 服务。跨院区或公网场景必须使用 TLS、访问审计、带宽评估和 TURN 策略。日志、配置和导出文件不得包含 LiveKit secret、HIS 凭据、FTP/SFTP/FTPS 密码或完整患者敏感信息。

## 全功能产品介绍版

以下段落用于网站产品页主体内容，覆盖当前开发文档和阶段计划中的全部功能点。对外发布时建议保留“规划支持、试点支持、可按项目交付”等状态词，避免把路线图误写成当前下载版已完整交付。

### 多路视频采集与手术室工作台

SmartST Lite 以手术室 Windows 桌面工作台为核心，规划支持 4 路本地画面预览，可接入 USB 摄像机、HDMI/SDI USB 采集卡、术野摄像机、内镜、超声、监护仪、PACS 工作站、显微镜等信号源。每路视频可以设置通道名称、角色、优先级和默认画面属性，常见角色包括全景、术野、内镜、医疗设备和辅助画面。

系统规划支持设备热插拔、设备占用提示、同型号采集卡重命名、通道角色持久化、分辨率和帧率降级策略。多路采集优先保证稳定性，不默认追求 4K 或最高帧率；4K、本地多路长期录像和特定采集卡组合应通过现场硬件白名单验证后再纳入交付范围。

### 音频设备与双向语音

系统规划支持板载麦克风、USB 全向麦、采集卡音频、外接声卡、会议音箱和扬声器设备选择，提供输入、输出、音量、电平、静音和设备状态提示。双向语音通过 LiveKit/WebRTC 体系实现，支持仅收看和交互模式下的不同音频权限。

全向麦、扬声器外放和回声消除效果依赖现场硬件组合，网站可宣传“支持 AEC/NS/AGC 音频策略和推荐硬件适配”，不应承诺所有麦克风与扬声器组合均可无调试稳定运行。

### PTZ 与设备控制

SmartST Lite 规划支持 USB、串口、VISCA 或厂商 SDK 等 PTZ 控制方式，可在设备支持时提供移动、停止、缩放、聚焦和预置位操作。系统设计要求先检测设备能力，再显示对应控制项；不支持 PTZ 的设备不显示误导性按钮。每个控制命令应有超时、停止和错误回显，避免云台持续运动或控制失效无提示。

### 呼叫、应答与互动模式

系统规划提供手术室端、示教室端和会议平板端之间的呼叫流程。示教室可向手术室发起呼叫，手术室可接受或拒绝；接受时可选择仅收看、交互或会议模式，并确定默认画面、允许通道、参与人数上限和远端权限。

业务服务负责端点注册、心跳、呼叫创建、应答、拒绝、挂断、房间创建和 token 签发。呼叫和权限不依赖 UI 假状态，所有关键动作应进入审计记录。

### LiveKit/SFU 实时互动

远程音视频通过 LiveKit/SFU 体系转发。手术室端默认只发布一路远程默认画面和手术室音频；示教室端订阅默认画面，可在交互模式下发布麦克风音频。远端如需查看第二、第三或第四路画面，应通过按需请求机制，由手术室端确认后再开放对应通道。

LiveKit 只负责房间、音视频转发、权限、Data/RPC 和可选 Egress，不承担采集卡驱动、PTZ 协议、录像资产管理、HIS 业务规则或患者信息管理。客户端不得保存 LiveKit API secret，所有短期 token 必须由服务端签发。

### Android 会议平板客户端

Android 会议平板规划作为正式客户端接入，不等同于手机 H5 观察端。会议平板可用于横屏大屏示教、移动会诊、会议讨论和交互控制，权限由业务服务 token 和手术室端接受呼叫时的策略共同决定。

会议平板能力规划包括仅收看、交互、会议模式、默认画面订阅、双向语音、弱网重连、横屏布局和独立人数上限。会议平板不应绕过业务服务直接加入房间，也不应直接连接手术室 Native Media Worker。

### 手机 H5 单向观察

手机端定位为 web-observer 单向观察端，不安装正式客户端，不参与双向互动。手机 H5 通过访问码或链接进入，由业务服务检查人数并签发只读 token，只允许订阅默认画面和手术室音频。

手机 H5 不允许发布麦克风、摄像头、标注、PTZ 或控制消息，不允许申请交互升级。手机观看并发由 LiveKit/SFU 转发，手术室端对同一默认画面只发布一次；10 个手机同时观看时，手术室上行不应按 10 路增长。

### 录像、回放、导出与上传

SmartST Lite 规划支持按通道录像、合成画面录像、单路 MP4 录制、本地录像文件管理、回放列表、检索、移动存储导出和上传任务。录像元数据通过 manifest 记录，包括通道、设备、开始时间、结束时间、文件大小、校验、异常结束状态、患者绑定 ID、上传状态和后续 AI job 预留字段。

系统应支持磁盘空间预警、录像开始/停止/失败状态机、程序异常退出后的已写片段恢复、导出校验、上传重试、失败原因记录和人工重传。正式医疗录像不应只依赖 WebView2 MediaRecorder；可交付版本应由 Native Media Worker 承担采集、编码、录像和异常恢复。

### HIS 绑定与文件索引

院内试点阶段规划支持 HIS、手术排班、患者绑定和录像索引。系统可根据院方接口条件对接 HL7、FHIR、WebService、数据库视图或手工录入流程。无正式 HIS 接口时，手工绑定作为兜底方案。

患者信息只用于录像绑定、检索和院内授权显示。默认不向远端示教室展示完整身份信息，不把患者姓名、身份证号等敏感字段写入录像文件名、日志或导出路径。需要展示完整信息时，应有权限、开关和审计记录。

### 标注、多显示器和会议控制

系统规划支持手术室端画面标注、远端同步标注、多显示器独立窗口、可调整布局和会议主持控制。标注事件可通过 LiveKit Data/RPC 发送，远端在低延迟范围内渲染同样标注层。

多显示器能力规划包括单画面扩展到指定 Windows 显示器、网格布局、主辅布局、单画面全屏和布局保存。会议主持控制规划包括静音、移除、升级或降级互动权限、限制参会人数和控制远端可见通道。

### 安全、审计与部署

系统规划支持院内局域网优先部署，必要组件包括 Windows 客户端、业务服务、LiveKit/SFU、TURN、日志审计和可选上传服务。跨院区或公网场景需要 TLS、TURN、带宽评估、访问审计和权限策略。

审计范围应覆盖登录、端点注册、呼叫、接受、拒绝、房间 token 签发、录像开始/停止、导出、上传、删除、权限变更和患者绑定。部署交付应包含 LiveKit/TURN/业务服务安装说明、端口说明、Windows 安装包签名、崩溃日志、媒体诊断、网络诊断和验收测试报告。

### AI 处理预留

文件管理模块规划预留 AI job 状态接口，用于后续接入视频分析、内容摘要、结构化标注或院内业务处理流程。当前网站可宣传为“预留 AI 处理接口和任务状态”，不建议写成已经内置具体诊断或医疗 AI 能力。

### 高级网络视频兼容

ONVIF、RTSP、SRT 和传统网络视频源保留为高级兼容能力。它们适合已有网络摄像机、编码器或跨区域视频链路的项目场景，但不作为 SmartST Lite 新版本默认入口。网站主叙述应坚持 USB 采集优先，避免用户误以为产品仍以摄像机 IP、ONVIF 鉴权和 RTSP 地址配置为核心。

## 当前开发状态

当前仓库已完成新阶段 PoC 基线，包括：

- USB-first 手术室工作台 UI 骨架。
- 业务服务 PoC：端点注册、心跳、呼叫、房间、mock token、观察端人数限制。
- LiveKit UI PoC：桌面端手动连接面板、仅收看/交互模式边界、远端 track 挂载。
- 手机 H5 单向观察 PoC：web-observer 只订阅，不发布音视频或数据。
- Media Worker IPC PoC：JSON Lines 控制面、mock 设备、状态机和 synthetic publisher 状态。
- USB 设备预检 PoC：DirectShow 枚举和短时打开验证。
- 录像 manifest 与单路短时录像 PoC。

当前已进入真实 LiveKit JWT、Native Worker 和 4 路 USB 硬件验证阶段。4 路摄像头基础链路可并发打开，但目标 USB 采集卡现场压力验证尚未完成，因此网站不得写成“已完成 4 路 USB 采集卡稳定验收”。

网站可提前宣传下一阶段能力，但建议对应写法为：

- “规划支持 4 路 USB 采集卡接入和本地预览”，不写“已完成 4 路长期稳定验收”。
- “规划支持 LiveKit/SFU 远程互动和手机 H5 单向观察”，不写“当前下载版已完成生产级公网并发”。
- “规划支持 Native Media Worker 多通道录像”，不写“当前版本已满足医疗归档录像要求”。
- “院内试点支持 HIS/上传/审计对接”，不写“已适配所有医院 HIS”。

## 版本路线

### 技术验证版

验证 Windows 桌面端能完成本地多路预览、一路远程互动、基础音频通话和单路录像 manifest 闭环。该阶段不承诺长时间医疗录像、正式 HIS 对接、多院区部署或全部 USB PTZ 摄像机兼容。

### 可演示版

完成手术室端和示教室端的呼叫、应答、仅收看/交互模式、主画面远程观看、双向语音、单路录像和基础文件列表。

### 院内试点版

扩展到 4 路预览、按通道录像、患者绑定、导出、上传、标注同步、多显示器扩展、会议人数限制、Android 会议平板客户端和手机 H5 单向收看。

### 可交付版

完成现场硬件白名单、长时间录像、断网/断电恢复、审计、权限、部署脚本、运维文档和验收测试。

## 网站模块建议

### 模块一：产品定位

突出“USB 采集优先”“Windows 桌面客户端”“轻量化手术示教直播”。不要再把 ONVIF、RTSP、摄像机 IP、编码器推流配置作为首屏卖点。

### 模块二：典型链路

展示链路：

```text
USB 摄像机 / USB 采集卡
  -> 手术室 Windows 客户端
  -> 业务服务 + LiveKit/SFU
  -> 示教室客户端 / Android 会议平板 / 手机 H5 观察端
```

### 模块三：核心功能

围绕本地预览、默认画面、远程互动、手机单向观察、录像导出、权限审计、患者信息脱敏展开。未交付能力可以提前宣传，但建议使用“规划支持”“试点阶段”“可按项目交付”表述。

### 模块四：部署方式

强调院内局域网优先部署，跨网段或公网场景需要 TLS、TURN、访问审计和带宽评估。可以宣传公网或跨院区能力规划，但不要写成“纯客户端即可稳定公网多人观看”。

### 模块五：适用与不适用

适用：教学示教、观摩、术中讨论、院内培训、轻量级远程观看。
不适用：诊断决策、治疗控制、生命支持、麻醉监护、医疗器械控制、完整数字化手术室平台替代。

## 旧版文案替换要求

必须删除或降级以下旧口径：

- “ONVIF/RTSP 摄像机转播系统”作为主定位。
- “输入摄像机 IP、账号、密码后开始转播”作为默认流程。
- “无需服务端即可公网多人观看”。
- “手机端可参与互动、标注、PTZ 控制”。手机端只能写单向观察；互动应留给示教室客户端或会议平板。
- “已完成 HIS、AI、FTP、Android 会议平板、多通道医疗录像”。这些能力可以宣传为规划、试点或项目交付能力。
- “4 路 4K 或多路长时间稳定已验收”。可以写“规划支持多路 USB 采集和现场硬件白名单验证”。

推荐替换为：

- “USB 采集优先的桌面版手术示教软件”。
- “通过 USB 摄像机和 HDMI/SDI USB 采集卡接入术野、内镜、监护仪等画面”。
- “远程观看由业务服务和 WebRTC/SFU 媒体中继承担”。
- “手机 H5 为单向观察端，不参与控制和互动”。
- “HIS、录像上传、Android 平板和多通道录像按院内试点路线推进”。

## 对外免责声明

SmartST Lite 用于手术示教、观摩、互动沟通和录像管理辅助场景，不作为诊断、治疗决策、生命支持、麻醉监护或医疗器械控制系统。涉及患者信息、手术视频、录像导出和系统日志时，应按院内敏感数据要求进行权限控制、脱敏、审计和存储管理。
